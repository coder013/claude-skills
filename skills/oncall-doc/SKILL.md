# OnCall 문서화

팀 온콜 처리 내용을 분석하여 재발 시 빠르게 대응할 수 있는 가이드 문서를 작성한다.

**저장 경로**: `/Users/jrlee/dev/repo/coder013/devlog/OnCall/`

---

## 실행 절차

### Step 1 - 온콜 내용 파싱

입력에서 추출한다:
- **제목**: `[...]` 형식에서 괄호 제거한 텍스트
- **식별자**: 주문번호, ID 등 구체적인 케이스 값
- **오늘 날짜**: `Bash(date +%Y-%m-%d)` 로 확인
- **SQL 분류**:
  - SELECT (주석 아닌 것) → 진단 쿼리
  - UPDATE / DELETE / INSERT → 처리 쿼리
  - 주석 처리된 SELECT → 추가 진단 옵션

### Step 2 - DB로 데이터 검증 (READ-ONLY)

MySQL MCP 또는 직접 쿼리로:
1. 진단 SELECT 실행: 핵심 진단 쿼리로 현재 데이터 상태 확인
2. 처리 흐름 검증: UPDATE 대상 ID가 실제 존재하는지 확인
3. UPDATE/DELETE는 runbook에만 문서화, 실제 실행은 수동으로

### Step 3 - 파일 경로 결정

파일명은 **제목 그대로** 사용한다.

기존 파일 존재 여부를 Glob으로 확인:
- **있으면**: 처리 이력 테이블에 행만 추가 (or 섹션 업데이트)
- **없으면**: 신규 생성

### Step 4 - 문서 작성

**문서 구조**:

```markdown
# {이슈 제목}

## 처리 이력

| 날짜 | 케이스 | 처리자 |
|------|--------|--------|
| YYYY-MM-DD | {식별자} | {담당자} |

## 원인

## 진단 쿼리

​```sql
-- 현황 확인
SELECT ...
​```

## 처리 쿼리

​```sql
-- 실행 전 반드시 진단 쿼리로 대상 확인
UPDATE ...
​```

## 주의사항
```

**SQL 재구성 원칙:**
- 원본 SQL을 그대로 보존, 단계별 처리 흐름으로 재배열
- 각 쿼리 앞에 "왜 이 쿼리를 실행하는가" 한 줄 설명 추가
- 처리 순서는 역방향 의존성 순 (자식 테이블 → 부모 테이블)
